# База данных

Классическое хранение БЧ в виде индексированной базы данных на основе key/value встраиваемых БД вроде LevelDB, или внешних типа MongoDB, когда каждый узел хранит свою собственную копию тезнически не представляется интересным для обсуждения.

Значительные же возможности открывает хранение БЧ на децентрализованной сети [IPFS](https://ipfs.io/ipfs/QmR7GSQM93Cx5eAg6a6yRzNde1FQv7uL6X1o4k7zrJa3LX/ipfs.draft3.pdf). В кратце IPFS представляет из себя направелнный ацикличный граф DAG, со ссылками в виде хэшей, отдельные части которого возможно разрозненно, находятся на узлах-участниках сети, ротинг и нахождение частей в котором осуществляется при помощи Kademlia-like протокола. 

Предлагается, что узлы БЧ не хранят блоки локально, а ведут свою цепочку в Меркель графе IPFS. Помня о том что для каждого узла экземпляр БЧ не идентичен другому, а изоморфен в ключе причино-следственных связей, образемый граф какраз и будет отражать эту струкутру. Каждый новый узел, после валидации, начнет свою цепочку от произвольного другого узла, и, вообще говоря может проивольно бросать свою и начинать заново от любой другой, по необходимости.

При использовании таких подходов избыточность (т.е. суммарное количество копий подтвержденной транзакции) может стать гораздо меньше чем при хранении всех транзакций на каждом узле (или в каждой цепочке графа IPFS). Роутинг IPFS в свою очередь позволит найти любую подвержденную тарнзакцию. Для дополнительной надежности можно ввобдить узлы хранилища, которые не будут заниматься обработкой транзакций, а только хранить части БЧ.

Таким образов огромное количество ресурсов для требуемое для хранения и обработки узлами БЧ 5 млрд. человек, описанные в предыдущем разделе становистя значительно меньше, а сама система приобретает дополнительнную децентрализованность.
